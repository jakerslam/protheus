# Workflow structure and level tracking

This directory treats workflow JSON as the source of truth for orchestration behavior. Rust code may load, validate, execute, and export traces, but it should not silently hardcode interaction behavior that belongs in a workflow CD.

## Workflow levels

Workflow levels are tracked in `workflow_composition.primitive_level`.

- Level `0`: primitive workflow. It does not call child workflows.
- Level `1`: composite workflow that calls only level `0` workflows, or a basic composite/template wrapper with no child workflow IDs.
- Level `N`: composite workflow whose highest child workflow level is `N - 1`.

When a workflow declares `workflow_composition.composed_of_workflow_ids`, its level must equal:

```text
max(child.workflow_composition.primitive_level) + 1
```

Adding a new level `0` primitive to a higher-level composite does not raise the parent level unless the parent previously had no higher-level children.

## Primitive composition doctrine

Workflow levels must compose monotonically. Higher-level workflows may call lower-level primitives, add routing around them, and define stop/checkpoint policy for their own level, but they must not make lower-level primitives narrower, slower, more brittle, or benchmark-specific.

If work on a higher-level workflow breaks a lower-level workflow or eval, treat that as a composition-boundary failure before treating it as a local prompt failure. The correct repair is to isolate the new behavior behind the owning composite workflow, a new primitive workflow, or an explicit lane, not to let level-specific behavior leak into shared primitive execution.

Production workflows must also follow the repo-wide Primitive-First System Doctrine. Do not hardcode behavior for a specific prompt, fixture, benchmark case, app shape, language, or framework inside shared workflow/runtime behavior. Specific-case hardcoding belongs only in eval or test fixtures. Reusable behavior belongs in primitives, composite routing, declared lanes, tool contracts, schema validators, safety policies, or extensible data/config contracts.

Repo-wide parent doctrine: `docs/workspace/primitive_first_system_doctrine.md`
Primitive registry policy: `docs/workspace/primitive_capability_registry_policy.md`

## Required tracking surfaces

Every workflow addition should update:

- `orchestration/src/control_plane/workflows/workflow_registry.json`
- The workflow JSON file under the correct `official/` or `lab/` path
- Parent workflow `workflow_composition_contract.child_workflow_calls` when the workflow is invoked as a child
- Parent workflow `workflow_composition.composed_of_workflow_ids` when it is part of the composition graph
- `local/state/ops/orchestration/workflow_composition_ledger_current.json`, regenerated by the workflow contract guard

## Primitive workflow format

A primitive workflow should include:

- `name`
- `workflow_type`
- `workflow_role`
- `description`
- `stages`
- `workflow_composition` with `primitive_level: 0` and no children
- `final_output_contract`
- `typed_execution_contract`
- `workflow_source_of_truth_contract`
- `interaction_gate_contract`
- `workflow_composition_contract` with `cd_kind: primitive` and no child workflow calls

## Composite workflow format

A composite workflow should include:

- `workflow_composition_contract.child_workflow_calls`
- `workflow_composition.primitive_level`
- `workflow_composition.composed_of_workflow_ids`
- Explicit child input/output artifact contracts
- A terminal artifact policy that returns only refs for child internals

Composite workflows should call reusable primitives instead of inlining broad reusable behavior. For coding workflows, context-pack construction belongs in `local_context_pack_builder`; implementation remains owned by the local coding execution workflow.

## Enforcement

The workflow contract guard validates registry shape, workflow source paths, composition metadata, and level consistency. Regenerate the composition ledger with the existing guard whenever workflow composition changes.
